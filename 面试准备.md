## 第〇部分：开场白与 JD 对位

### 30 秒电梯陈述（背熟）

> 我做了 OWCopilot——面向开放世界游戏内容生产的本地优先 AI 工作台。它解决的不是"让 AI
> 写任务"，而是策划真正的痛点：**设定散落、改动牵连不可见、AI 产物不可信**。核心是
> "内容图谱 + 26 条确定性一致性审计 + 带引用的检索问答 + 受约束生成 + 非结构化文稿提炼"，
> 所有 AI 产物带溯源进人审队列，人审是唯一落盘通道。工程上 381 个离线测试、五道 CI 门禁、
> Docker 一键起 API+UI，多轮真实 LLM 验证总成本不到 2 美分。

### 1 分钟版本（追加）

> 架构上四端一核：CLI、REST、MCP、Streamlit 工作台共享同一套 pipeline 层。LLM 接入收敛在
> 一个网关单点，缓存（精确+语义两级）、级联路由、遥测计费全挂在网关上，基准实测降本 72%。
> 两个最能体现 Agent 工程的设计：一是**影子校验**——模型提的修复补丁先在世界副本上复跑
> 全量审计，会引入新错误的候选在人看到之前就被丢弃，这条防线在真实测试里真的拦下过坏候选；
> 二是**文稿提炼的人机协同闭环**——丢一本小说进来，多块并行抽取、按名合并、生成人物关系图
> 和剧情节拍，**缺什么字段就生成缺口清单**，用户逐项填或一键让 AI 补一版再确认，最后整包
> 过人审落盘。

# 第一部分：开发者视角讲解（面向初学者）

## 1. 项目要解决什么问题

### 1.1 行业背景

开放世界游戏（《巫师 3》《原神》《荒野大镖客 2》这个量级）的内容量极大：成百上千个 NPC、
任务、地点、阵营、术语。文案/剧情策划的日常痛点：

1. **设定散落**：世界观写在 Word/飞书/Excel 各处，"某角色到底是哪个阵营的"要翻半天；
2. **一致性崩坏**：任务 A 说某角色已死，任务 B 里他又活了；时间线、阵营、地理互相矛盾；
3. **改动牵连不可见**：删一个 NPC，哪些任务、对白、本地化文本会坏？靠人脑记不住；
4. **AI 不可信**：直接用聊天 AI 写任务，会引用不存在的角色（幻觉）、违反世界观、语气出戏，
   而且没人知道哪些内容是 AI 写的、有没有人审过；
5. **存量资产难迁移**：积累的小说设定稿、剧本、散乱笔记没法直接变成可管理的结构化资产。

### 1.2 我们的回答（产品定位）

OWCopilot 是一个**内容中枢（content hub）**：

- **把设定变成结构化资产**：实体/关系/任务/区域/术语/对话树进入统一内容仓与图谱；
  标准策划表（Excel 多 Sheet/JSON/Luban/Markdown）直接导入，**非结构化文稿（小说/剧本/
  笔记，含 .docx）由提炼管线转成生产级草案**；
- **把一致性变成可执行的检查**：26 条确定性规则，逐条带证据，零模型成本；
- **把"问设定"变成有引用的检索问答**：答案必须引用图谱条目，查不到就明确拒答；
- **把 AI 限制在安全的笼子里**：生成物只能引用图谱内实体、生成即审计/lint、强制进人审
  队列，人审通过才落盘，origin（人写/AI 草稿/AI 修复）永久留痕。

一句话："**查设定有出处，跑审查有证据，AI 产物必过人审**"。

### 1.3 工程叙事：从 v1 到 v2 的转向（面试加分点）

第一阶段（P0–P3）做的是"LLM 生成任务 → 校验 → 落进 UE5/Unity"的生成管线，跑通了
LangGraph 编排、成本工程、引擎适配。复盘发现：**生成不是策划最痛的点，"信不过"才是**。
v2 把重心转向"确定性审计 + 可溯源问答 + 人审门"，生成降级为受约束辅助；之后再补上
创世/提炼这类"冷启动"能力。这个 pivot 主动讲，它展示产品判断力。

## 3. 项目结构（怎么分层、为什么）

```
src/owcopilot/
├── cli/  service/  mcp_server/  app/      # 四个"端"：CLI / REST / MCP / Workbench UI
│       （app/: dashboard.py 纯 UI 壳；actions.py + view_models.py 不依赖 Streamlit）
├── pipeline/                              # 工作流编排：project/audit/patches/review/ingest
├── content/  graph/  audit/  impact/      # 领域层：内容模型/图谱/审计/影响
├── retrieval/  qa/                        # 检索与问答
├── patches/  assist/  worldgen/           # 修复闭环 / 辅助生成（草稿/台词/对话树/物案）
├── extraction/  inspiration/              # 文稿提炼 / 灵感参考库
├── exporters/  adapters/                  # 导出与引擎桥
├── llm/                                   # 网关/路由/缓存/遥测 —— 模型调用唯一通道
├── core/                                  # LangGraph 编排器 + 状态 + 协议
├── storage/  trust/  telemetry/           # SQLite 运行库 / 溯源安全 / 成本核算
├── evaluation/  fakes/                    # 评测基准 / 测试替身
tests/        # 381 个测试，全部离线、$0、确定性
```

**分层讲法（背熟）**：端层只做协议转换；pipeline 层编排"打开项目→执行→持久化"；领域层
纯函数+pydantic，不碰网络；`llm/` 是唯一外部依赖点——假 provider 一替换，整个系统离线
测试，这就是 381 个测试全 $0 的原因。

**两类真相分离**：内容仓（文件，git 友好，人可读）是"世界的真相"；SQLite
（`.owcopilot/runtime.sqlite`）只存运行态（问题/补丁/待审队列/参考索引）。删掉 SQLite
不丢设定，重跑审计即可重建。

## 4. 核心功能实现细节

### 4.1 内容接入与图谱

- 导入器按格式拆分，共享 `FieldMapping`（中文表头"名称"→name、别名列、默认值）；冲突默认
  跳过并记录；**默认 dry-run**，UI 里是"预览→确认写入"两步。国内策划的常见产物——Excel
  多 Sheet 策划表、Luban 表、飞书/Word 导出（docx 走提炼通道）、Markdown 设定页——都有
  对应入口。
- 图谱：networkx 有向多重图，节点 `entity:/quest:/region:` 等，边=关系+结构引用；时间线
  单独索引供时间悖论规则用。

### 4.2 确定性审计（项目的脊梁）

- 规则是纯函数：`AuditContext`（bundle+graph）进、`list[Issue]` 出，每条带 rule_code、
  severity、target_ref、message、**结构化证据**。
- 26 条规则八组：引用完整性、图谱（孤儿/环/重复）、世界观（时间线/术语/角色状态）、区域
  约束、管线（本地化键/超长/占位符）、安全（注入痕迹）、信任（AI 未审）、**对话树**
  （断链 DIALOGUE_TREE_BROKEN_LINK / 未知说话人 / 不可达节点）。
- **baseline 棘轮**：存量项目第一次接入冒出的历史问题存成指纹集合，此后只对新增报错——
  从 CI 工具（ESLint ratchet）借的落地经验。

### 4.3 检索与引用问答（迷你 RAG，每一步可解释）

- 三路召回：**BM25**（CJK 二元组 fallback 解决中文分词）、**向量**（默认哈希词袋，可注入
  真嵌入）、**图扩展**（命中实体沿边扩一跳）。
- **RRF** 合并 → **token 预算器**截断 → `ContextPack`（带 refs 白名单）。
- QA：模型输出 JSON（answer/citations/mentioned_entities）→ **verify**：引用必须在白名单
  内（裸 ID 规范化）、提到的实体必须能解析，否则记 unresolved；召回不足→显式拒答。

### 4.4 修复闭环（影子校验，必讲）

```
audit 发现 issue → suggest：确定性修复器($0) + LLM 候选(可选)
   → 每个候选在影子副本上应用 → 复跑全量审计
   → 引入任何新 error 的候选直接丢弃（用户根本看不到）
   → 人工 apply（记操作者）→ 自动复跑 → rollback 可逆
```
**真实战绩**：第三轮真实 LLM 测试中，模型给 TIMELINE_VIOLATION 提的修复被影子校验拦截
（方案引入新错误）。该类问题无确定性修复器，被诚实标为"人工处理"。防线被真实模型验证过。

### 4.5 受约束生成与人审门（四种创作物）

- **任务草稿**：brief + ContextPack → JSON → 宽容 parser（容忍 rewards null/{}、字段别名
  等真实观测到的形态漂移）→ 生成即审计 → 待审队列。
- **台词批量**：每个说话人组装 **voice card**（名字/描述/阵营/语气/禁忌）→ 生成 → lint
  （长度/禁用词/实体白名单），不合格直接过滤并标注原因。
- **分支对话树**：参与者 voice cards + 简报 → JSON 树（nodes/choices/next）→ 解析时
  ID 规范化、说话人对齐 → **结构完整性检查**（断链/未知说话人/不可达）→ lint → 待审；
  采纳落盘后审计规则持续看护；UI 里用 graphviz 渲染树形结构。
- **物案批量**（物品/技能/成就）：名称清单 + 主题 + 风格圣经上下文 → 每项
  {功能描述+风味文本} → lint → 待审 → 采纳后写成对应类型实体。
- **review 是唯一落盘通道**：七种待审类型（草稿/台词/补丁/世界草案/提炼草案/对话树/
  物案批次）各有校验后的写入路径，操作者署名入审计日志。

### 4.6 一键创世 + 灵感库

- 灵感库：上传/粘贴参考 → 切片（1800 字符/160 重叠）入 SQLite 索引；**与项目事实检索
  物理隔离**，保证参考永远不会被当成世界观事实引用。
- 创世：表单 brief → 同时拉项目事实包（可选）与灵感包 → 模型输出结构化世界 → 服务端
  强制收敛（ID 规范化、缺额补齐、关系去重、未知引用回退）→ 整包待审；附
  **reference_report**（每份参考用于什么、如何转化、排除了什么）——版权与原创性的工程化表达。

### 4.7 导出与引擎适配

generic（JSON 全集）/ unreal（DataTable CSV + 本地化 CSV）/ unity（每任务 camelCase
JSON + index）。manifest 给每个产物记 sha256；导出路径过 `resolve_under_root` 防穿越。

### 4.8 文稿提炼：非结构化 → 生产级（Agent 工作流的招牌案例）

这是"把一本书变成可管理资产"的完整管线，面向 Agent 岗时重点讲：

```
原稿(txt/md/docx) → 分块(3500字/200重叠)
  → 逐块 LLM 抽取（characters/locations/factions/items/terms/relations/beats，JSON）
  → 跨块按名合并（描述取最长、别名/特质并集、关系去重）
  → ID 规范化（CJK 保留 slug + 冲突自动编号）、关系端点名→ID 解析
  → 产出：候选 ContentBundle + 人物关系图 + 剧情节拍(beats) + 缺口清单(gaps)
  → 缺口环节（HITL 核心）：每个缺失字段生成一个具体问题
       ├─ 用户逐项手填
       └─ 或一键"AI 补全"→ 建议写进 gap.suggestion → 用户仍要确认
  → 提交（可选：节拍转任务骨架）→ 草案审计 → 审阅台 → 人审采纳才落盘
```

设计要点（面试官会追问的）：
- **为什么分块 3500 字**：单块要装进 cheap 模型上下文且留输出余量；段落边界切分 + 200 字
  重叠防实体在边界被切断。
- **跨块合并策略**：名字做 key；描述"取最长"（信息量代理）；这是确定性合并，不再花一次
  LLM 调用做"汇总"，省钱且可测。
- **关系端点解析**：模型输出的是名字，服务端映射到规范 ID；解析不了的进
  `unresolved_relations` 单独展示，不污染图谱。
- **缺口而不是编造**：抽取 prompt 明确"原文没给就留空"，空字段变成 gap 问题——把幻觉
  压力转化为人机协同任务。AI 补全也只写进 suggestion，确认权在人。
- **docx 解析零依赖**：docx = zip+XML，用标准库 zipfile + ElementTree 抽 `w:t` 文本，
  不为一个格式引第三方包。
- 离线确定性 provider（正则启发式）让整条管线在 CI 里 $0 跑通——测的是**管线行为**，
  抽取质量由真实模型 + 评测兜底。

### 4.9 两个"产品工程"故事（行为面常用）

**BYO API Key 的信任设计**：要让普通创作者敢把自己的 Key 填进来，靠的不是承诺而是架构事实——
本地优先意味着 Key 只进当前进程内存（不落盘除非用户选择）、调用直连模型厂商、零遥测。产品层
把它做成体验：服务商预设（DeepSeek/OpenAI/Kimi/智谱/自定义 Base URL）→ 粘贴 Key →
**连接探活**（最小完成请求，回显延迟或分类后的中文错误：401/429/超时/网络）→
**会话成本熔断**（达到自设上限，真实模式按钮全部锁定）。面试句式："信任 = 架构事实 + 可验证的
反馈 + 不可逾越的护栏，三件套缺一不可。"

**在 Streamlit 里做游戏式新手引导**（框架约束下的工程创造力）：Streamlit 没有任何 DOM 级引导
能力。解法链条：① Streamlit 组件 iframe 与主页面**同源**，注入脚本可以拿到
`window.parent.document`；② 高亮用一个带 `box-shadow: 0 0 0 9999px rgba(...)` 的定位环实现
"镂空变暗"，零依赖；③ 关键洞察——**Streamlit 的标签页切换是纯客户端行为，不触发脚本重跑**，
所以引导可以自己点击标签页带用户逛完全站；④ 降级策略：锚点选择器找不到就自动跳到下一步，
ESC 随时退出；⑤ Python 侧只在点击"新手引导"按钮的那一次 rerun 注入组件，不污染常规渲染。
面试句式："框架的边界不是产品的边界——前提是你知道边界在哪、为什么能绕过、绕过的代价是什么。"

## 5. Agent 设计细节

**先定义**：感知（读状态）→ 规划 → 行动（调工具）→ 观察反馈 → 修正。OWCopilot 是
**受约束的单 Agent + 多条专用工作流**，而非自治多 Agent：

- 核心环（LangGraph）：PLAN（便宜模型分解意图）→ EXECUTE（生成，强模型档）→ VERIFY
  （确定性验证器，**工具即真相**）→ REPAIR（确定性优先，限次 `max_repair_attempts=2`）
  → 回 VERIFY；超限 → 失败交人工。
- 专用工作流复用同一哲学：修复闭环的"影子校验"、提炼管线的"缺口环节"，本质都是
  **生成 → 机器可判定的验证 → 人终审**。

**关键立场（必背）**：
1. **为什么不做自治多 Agent**：游戏内容是高一致性领域，错误代价是世界观崩坏。自由度=
   风险面。把"自由"换成"可验证"：模型只在固定几个口子说话，其余全是确定性代码。
2. **反馈信号从哪来**：不是模型自评（自评会说谎），是确定性验证器的 error 列表——相当于
   给 Agent 一个"编译器"。
3. **人在哪里**：人审是终审。Agent 的所有产物只能停在待审队列，这是治理边界；提炼缺口的
   "AI 补全后仍需人确认"是同一原则的细粒度版本。

## 6. LangGraph 设计细节

代码在 `core/orchestrator.py`，刻意薄用：

```python
g = StateGraph(TaskState)            # TaskState: TypedDict（intent/plan/artifact/
g.add_node("plan", plan_node)        #   issues/repair_attempts/max_repair_attempts/log）
g.add_node("execute", execute_node)
g.add_node("verify", verify_node)
g.add_node("repair", repair_node)
g.set_entry_point("plan")
g.add_edge("plan", "execute")
g.add_edge("execute", "verify")
g.add_conditional_edges("verify", route_after_verify,
                        {"done": END, "failed": END, "repair": "repair"})
g.add_edge("repair", "verify")
return g.compile()
```

- **状态**：TypedDict 增量更新（节点返回 dict 补丁，LangGraph 合并）；`log` 累积轨迹。
- **条件路由**：`route_after_verify` 看 error 数与尝试次数，三出口，终止条件显式。
- **依赖注入**：节点闭包持有 gateway/generator/validators/repair_strategy，全是 Protocol；
  测试注入假实现，图接线零改动。
- **为什么 LangGraph 而非手写**（诚实版）：这图手写 50 行也行。买的是：声明式拓扑（评审
  一眼看懂环与出口）、状态合并语义、未来 checkpoint/人工中断不用重写。同时隔离在 core/
  一层，要摘除半天搞定——"用框架但不被绑架"。

## 7. LLM 工程：网关、路由、缓存、成本

### 7.1 网关单点（`llm/gateway.py`）

所有模型调用必经 `LLMGateway.complete(task, system, user)`：缓存、路由、遥测、重试挂在
这一个方法上，加新能力时调用方零改动。Provider 是 4 行协议，第 4 个返回值承载 DeepSeek
服务端前缀缓存命中数（按便宜价计费）。JSON mode 门控在"prompt 含 json"上（否则 400）；
超时与 `max_tokens`（默认 3000）可配；重试带失败分类（timeout/rate_limit/auth/connection）。

### 7.2 路由

StaticRouter（任务→档位）+ **CascadeRouter**（FrugalGPT 思路）：generate 先走 cheap，
**置信信号 = 复用确定性验证器**——cheap 产物过不了校验，包装器以显式 `tier=strong` 重发。
"便宜优先，破坏世界观才升级"，不需要额外打分模型。

### 7.3 缓存（三层叙事）

| 层 | 机制 | 省什么 |
|---|---|---|
| L1 Exact | sha256(tier+system+user) | 整次调用（$0） |
| L2 Semantic | 嵌入最近邻，余弦≥0.9，**scope=tier+system** | 换说法的重复请求 |
| 服务端前缀缓存 | DeepSeek prompt cache（命中价≈1/50） | 共享前缀输入 token |

L2 两个关键设计：**只嵌 user 消息**（system 是检索出的设定，又长又同质，嵌入会误并）；
**scope 限定 tier+system 相同**（不同上下文绝不互相命中）。LayeredCache 把 L2 命中提升进
L1；空响应不缓存。REST 层共享应用生命周期缓存，重复问答 $0（测试钉死 hit_rate==1.0）。

### 7.4 成本核算与护栏

每次调用记 CallRecord → 按档位单价折算 USD → 工作流汇总 cost_budget；**所有 LLM 入口
接受 max_cost_usd**；UI 每个操作回显本次+会话累计。实测：缓存+级联基准降本 **72%**；
多轮真实验证成本 $0.0045 / $0.0057 / $0.0071。

## 8. 上下文、记忆、存储

- **上下文 = ContextPack**：三路融合 + 预算截断 + refs 白名单。不用长上下文塞整个世界书的
  三个理由：成本线性涨；中部信息丢失（lost in the middle）；**白名单是引用校验的前提**——
  塞了全世界就没法判定引用合法性。
- **记忆分层**：长期事实=内容仓文件（git 友好可评审）；运行态=SQLite（WAL+busy_timeout+
  索引）；会话=session_state（问答历史、会话成本，刻意不持久化）。
- 没做对话式长期记忆（mem0 类）：产品形态是工作台操作而非聊天陪伴，记忆需求由结构化存储
  满足——这是边界，主动说。

## 9. 容器化与上线

- **镜像**：多阶段构建（builder 装 `.[serve,app,live]` 进 venv，runtime 只拷 venv，
  python:3.12-slim）；非 root（uid 10001）；HEALTHCHECK 用标准库打 `/health`；
  **不带测试、不带密钥**。
- **编排**：一镜像两角色——api(:8000) + ui(:8501，挂载内容仓)；可选 redis 块（多副本共享
  缓存与限流）注释就绪；`.env` 被 .dockerignore，运行时注入。
- **上线安全清单**：real 模式双重 fail-closed（无 OWCOPILOT_API_KEY→403，无 provider→503）；
  限流（内存窗口/redis 可切）；导出路径防穿越；World Bible 输入预算（20 万字符/500 实体）
  + 注入扫描；已知边界——Streamlit 定位单团队内网，多租户换前端（API 已就绪）。

## 10. 质量与评测体系

- **381 个测试**全离线/$0/确定性（假 provider），含 Streamlit AppTest 渲染测试；
- **五道门禁**进 CI + pre-commit：ruff check / ruff format / mypy（150 文件）/ pytest / eval；
- **验收基准**：程序化生成双语世界（65 实体/10 区域/36 任务），**埋 25 个分类错误**——
  干净世界 0 误报；检出 25/25（门槛≥85%）；影响召回 100%；30 条双语检索 30/30（门槛≥90%）；
  QA 有据答/拒答全对。**评测先于功能**：每条规则先定"该抓什么"再写实现。
- 真实 LLM 验证 = 手动 GitHub Actions workflow（花钱的事不进每次 CI），报告归档。

# 第二部分：面试官拷问与应答

> 姿态：给结论 → 给理由 → 给边界/代价（诚实）。

## A. 技术选型辩护

**Q1：为什么 SQLite 不上 PostgreSQL？**
> 本地优先单团队工作台，SQLite 存的是可重建的运行态（核心资产在文件仓里 git 管理）。
> 做了 WAL + busy_timeout + 索引撑并发读写。升级路径清晰：存储层独立模块，换 PG 只动一层。
> 没有多租户写并发前上 PG 是过度设计。

**Q2：为什么不用向量数据库？**
> 检索对象是单项目千级设定条目，线性扫描+RRF 已拿到 30/30 命中。向量库解决的是亿级 ANN，
> 瓶颈不在那。嵌入器与索引都是协议注入，规模到了换实现即可。先量化需求，再上基础设施。

**Q3：为什么前端 Streamlit 不写 React？**
> 用户是策划，要的是今天就能用。Streamlit 让 UI 成为纯壳——业务全在 Streamlit-free 的
> actions/view_models（可单测，CI 跑 AppTest）。多租户需求出现时，React 对接的是已资源化的
> REST API。"UI 可替换"是架构事实，不是口头承诺。

**Q4：为什么 LangGraph？是不是简历驱动？**（诛心题，要诚实）
> 这图手写 50 行也能做。选它买三样：声明式拓扑（环路与出口在评审里一眼可见）、状态合并
> 语义、为 checkpoint/人工中断留路。同时只在 core/ 一层用，领域零依赖，半天可摘除。
> "用框架但保留摘除能力"比"手写一切"或"全盘框架化"都成熟。

**Q5：为什么自建缓存/路由，不用 LangChain 全家桶？**
> 缓存与路由是核心差异化逻辑（L2 scope 设计、验证器当级联置信信号），不是通用件。自建
> 几百行全覆盖测试，每个行为可解释可调参；通用框架在关键决策处给的是黑盒最大公约数。

**Q6：你的"向量检索"是哈希词袋，不算真向量吧？**
> 对，默认实现为了离线 $0 测试。接口是 Embedder 协议，生产注入真嵌入即可。工程点：
> **嵌入质量是可替换插件，三路召回+RRF+预算的架构才是不变量**；质量由验收基准命中率兜底。

**Q7：创意生成也走缓存，同 prompt 永远同结果合理吗？**
> 刻意取舍：工作台的重复请求多来自误触发/复跑，缓存省这部分；要"换个版本"用户会改 brief
> （一个字也 miss L1，L2 还有 0.9 阈值）。需要同 brief 多样化时，把变体序号编进 user 消息
> 或对创意任务关缓存——开关在网关，一行配置。

## B. 系统设计与扩展性

**Q7.5：你有个"离线模式"，为什么产品界面上看不到？**
> 这是一次真实的产品修正：离线确定性应答器本质是**测试基座**（让 381 个测试 $0 跑、CI 不依赖
> 网络），早期被顺手暴露成了 UI 开关——但对终端用户它毫无意义（"离线生成"给出的是确定性
> 占位内容）。后来把它从界面撤下：未接入 Key 时确定性功能照常可用、创作功能落锁并引导接入；
> 离线 provider 退回测试与 CLI/REST 的开发者面。教训一句话：**开发者工具和产品功能要分开，
> "我调试时好用"不等于"用户需要看到"。**

**Q8：内容规模到 10 万实体怎么办？**
> 四步：审计增量化（影响分析就是现成依赖图）；检索换真嵌入+ANN；SQLite→PG、图谱按需加载；
> 审计规则并行化（规则间无共享状态）。**每层都是协议，替换不动调用方**。

**Q9：多人同时编辑怎么办？**
> 当前边界：单团队 + git 工作流（分支/PR 是天然协作锁），运行库 WAL 并发读。真多写者：
> 内容服务化（API ready）+ 乐观锁（content_hash 就是版本指纹）+ 审阅按人分配。能画演进图，
> 但当前用户故事没有它，不提前实现。

**Q10：模型 API 挂了系统什么表现？**
> 分级降级：网关重试（指数退避）→ 分类错误上抛 → 端层人话提示。**主干（审计/影响/检索/
> 导出）根本不依赖模型**；生成类离线模式还有确定性应答器兜底。AI 是增强，不是单点。

## C. LLM 质量问题排查（高频场景题）

**Q11：用户反馈"AI 输出质量不高"，怎么排查？**（背这个五层框架）
> 先定义"不高"：格式坏、事实错、风格不符、还是无聊？然后从下往上：
> 1) **解析层**：telemetry 看原始输出——JSON 截断（撞 max_tokens）？形态漂移没接住？
>    看 parse_failed 计数；
> 2) **上下文层**：复现该请求的 ContextPack——该给的设定检索到了吗？预算够吗？中文分词
>    走 CJK fallback 了吗？**大部分"模型不行"其实是"喂的上下文不行"**；
> 3) **模型层**：该升档了吗？级联触发了吗？frontier 复跑 A/B；
> 4) **Prompt 层**：约束明确吗？加 few-shot、把世界观规则写进 system；
> 5) **验收层**：都对但仍不满意 = 主观品味，人审打回 + 样本沉淀进评测集变成下轮门槛。
> 本项目每层有探针：telemetry、context-pack 命令、影子审计、待审通过率。

**Q12：幻觉（编造不存在的 NPC）怎么治？**
> 三道闸：生成前 prompt 只给白名单实体；生成后 verify 校验引用与提及；落盘前审计再扫。
> 提炼管线再加一道：**原文没给的字段不许编，变成缺口问题交给人**。思路不是求模型别编，
> 是**让编造无法通过机器校验**。

**Q13：没有 ground truth 的生成任务怎么评测？**
> 两类指标：客观代理（结构合法率、审计零新增 error、lint 通过率、引用合法率——验收基准
> 25 埋错全检出干的就是这个）；主观质量（人审通过率当北极星，驳回样本归因进 golden 集）。
> 金句：**"能自动判定的全部自动化，必须人判定的做成最低摩擦的人审流。"**

**Q14：线上成本暴涨怎么查？**
> telemetry 按 task/tier 聚合：哪个任务调用量/输出 token 涨了？缓存命中率掉了（内容大改会
> 全 miss）？重试循环（分类计数）？然后护栏复盘：max_cost_usd 阈值设了吗？输出 cap 被调大
> 了吗？预防面：token 上限默认 3000、重试上限、全入口护栏、UI 逐操作回显。

**Q15：JSON mode 返回不合法/截断 JSON？**
> 真实遇到过（超长草稿撞 max_tokens）。处理：宽容 parser（剥围栏、取首尾大括号）→ 仍失败
> 计 parse_failed 走重试 → 形态漂移逐项容忍（全部来自真实轮次观测，各有测试）。原则：
> **对外严格（schema 校验），对内宽容（解析容错）**。

**Q16：Prompt injection 怎么防？**
> 攻击面：上传的世界书/参考/手稿里埋"忽略以上指令"。防线：入库注入扫描+输入预算；
> **架构性防御**——模型输出永远不直接执行/落盘，只能变成待审队列里的结构化提案，注入成功
> 的上限是"生成一条会被人审驳回的草稿"；MCP 只暴露只读工具。
> 金句：**"我假设注入一定会发生，所以把爆炸半径压到人审队列为止。"**

## D. Agent 专题（Agent 研发岗加问）

**Q17：你的提炼管线算 Agent 吗？多步抽取为什么不用一个大 prompt？**
> 按"感知-规划-行动-反馈"定义，它是一条**固定拓扑的 Agent 工作流**：每块抽取是行动，
> 合并与缺口检测是确定性观察，AI 补全+人确认是带 HITL 的修正环。不用一个大 prompt 的原因：
> 1) 全书超上下文窗口，硬塞会丢中部信息；2) 分块+确定性合并把"长文理解"降为"短文抽取"，
> cheap 模型就够，成本线性可控；3) 合并/解析/缺口全是可测的确定性代码，质量问题能定位到
> 层，而大 prompt 是不可调试的黑盒。

**Q18：Human-in-the-loop 怎么设计才不烦人？**
> 原则：**人只做机器判不了的事**。机器能判的（结构、引用、长度、术语）全自动化；机器判
> 不了的（这个角色的动机写得对吗）做成最低摩擦交互——缺口清单预填 AI 建议，人改一下或
> 直接确认；审阅台一键采纳/驳回。反例是让人盯原始 JSON。指标：人审平均耗时和驳回率。

**Q19：多 Agent 协作（如生成者+评审者）考虑过吗？**
> 考虑过且部分实现了——但"评审者"不是另一个 LLM，是确定性验证器+影子校验。LLM 评审 LLM
> 有同源偏置（两个都会犯同类错），而我的领域恰好能写出确定性裁判，这比模型裁判强。如果
> 未来要评"文风是否出戏"这类主观项，会引入 LLM-as-judge，但只作为打分信号进人审界面，
> 不作为自动门禁。

**Q20：Agent 的可观测性怎么做？**
> 三层：LangGraph 状态里的 log 字段累积每步轨迹；网关 telemetry 记每次调用（task/tier/
> tokens/缓存/时延/成本）；业务层每个工作流返回结构化结果（候选数/淘汰数/parse_failed/
> 缺口数）。排查时先看业务计数，再下钻 telemetry，最后看原始输出。

**Q20.5：你的工具要服务非专业创作者，怎么做"开箱即用"？**
> 五件套：`owcopilot ui` 一条命令进工作台；游戏式新手引导代替文档（在操作位置旁弹卡片，
> 跨页带逛）；创世模板库（六种题材一键填表）；离线模式 $0 先体验全流程，再决定填 Key；
> 加载动画+逐操作成本回显消除"它卡了吗/花了多少"的焦虑。原则：**把"读文档"变成"被带着做一遍"**。

**Q20.6：写作者贴一段新章节，怎么知道它和设定打不打架？**
> 文稿体检：确定性 v1——已知名称/别名解析（最长优先）、术语禁用写法定位+标准名建议、
> 未知专名候选（引号/说话人/地名模式）。评测沿用埋错方法论：测试植入禁用词与陌生人名，
> 必须全部抓到，干净文本必须零误报。这是"同一套可信验证哲学迁移到新场景"的例子——
> 面试官问"你的方法论可复用吗"时就讲它。

## E. 行为面 / 复盘

**Q21：项目里最难的 bug？**
> 讲影子校验拦截的故事：真实模型给时间线矛盾提修复，方案本身引入新错误。难点在"怎么发现"
> ——不是人盯，是影子副本复跑全量审计的机制自动暴露。升华：**LLM 系统最值钱的投资是
> "机器能自动判错"的验证面**，它把模型质量问题从线上事故降级成日志里的一行 rejected_count。

**Q22：如果重做会改什么？**
> 1) 评测基准更早建（v1 期就该有埋错基准，pivot 会更快）；2) 更早抽 Streamlit-free 的
> actions 层（后来补的）；3) "参考与事实隔离"的概念应该 day-1 就有——它一出现，创世和
> 提炼的版权叙事立刻清晰。

**Q23：这个项目 AI 参与了多少？**（新时代必考）
> 诚实模板：架构决策、产品 pivot、验收门槛设计是我主导；实现大量使用 AI 结对——而项目
> 主题恰是"如何让 AI 产出可信"：每行 AI 代码过五道门禁+363 个测试+真实轮次验证。仓库里
> 任何设计决策我能从动机讲到代价，随便挑一处问。

## F. 通用概念题（用本项目素材回答）

- **RAG vs 长上下文**：§8 三点（成本线性/中部丢失/白名单可校验）。
- **Function calling vs JSON mode**：本项目用 JSON mode（单轮结构化输出、服务端 pydantic
  校验）；function calling 适合"模型决定调哪个工具"的多轮场景——我们的工具编排是确定性
  管线，不需要模型决策。
- **温度**：结构化生成低温求稳；多样性靠"变体序号+voice card 差异"进 prompt，而非拉高
  温度赌随机——可复现性优先。
- **微调 vs 提示工程**：当前量级 few-shot+检索上下文够用且零训练成本；若风格一致性成瓶颈，
  用人审通过样本做 SFT/LoRA——数据收集管道（审阅队列）天然存在。

## 11. LangGraph 设计答辩（必背三连）

**Q：为什么用 LangGraph，而不是手写循环 / CrewAW 类多智能体框架？**
A：我们要的不是"多个角色聊天"，是**带条件分支的状态机**：PLAN→EXECUTE→VERIFY→(通过 END / 失败且有额度 REPAIR→VERIFY)。LangGraph 给了三样刚需：显式 StateGraph（图即文档，新人看图就懂控制流）、共享类型化状态（各节点读写同一个 state dict，不靠参数透传）、条件边（`add_conditional_edges` 读 VERIFY 的审计结果决定走向）。手写循环这三样都要自己发明；多智能体框架则把控制流藏进"角色对话"里，可观测性反而塌了。同时坦白：LangGraph 只用在 `core/orchestrator.py` 这一处——其余链路（提炼五步、清查三层）是固定步骤，用普通函数编排。**该 agent 的地方 agent，能 workflow 的地方绝不 agent**——这句本身就是加分答案。

**Q：你的 VERIFY 为什么不用 LLM 当裁判？**
A：用确定性规则（26 条审计）当裁判，LLM 输出才有客观对错；LLM 评 LLM 会把幻觉链式放大，且不可回归测试。规则给出的是结构化证据（哪条引用不存在、哪两个任务时间线倒挂），REPAIR 节点能拿证据定向重写——比"再试一次"的盲目重试收敛快得多。

**Q：REPAIR 修不好怎么办？**
A：有限额度（防死循环烧钱），耗尽后带着违规清单进人审队列——失败的终点是人，不是无限重试。这是 HITL 的兜底语义。

## 12. Context 工程答辩

**Q：上下文怎么组装？**
A：`retrieval/context_pack.py`：BM25（CJK 退化分词）+ 向量 + 图谱邻接扩展三路召回 → RRF 倒数排名融合 → token 预算器截断 → 每条带 `[ref]` 标号。回答必须引用 ref，引用不出来就拒答——把"接地"做成了协议而不是祈祷。

**Q：讲一个你真实修过的 context bug。**（这题就讲第十二轮）
A：创世表单曾把整个 brief `model_dump_json()` 直发模型——空的 `"player_fantasy": ""` 字段名本身成了"请发明主角"的暗示，用户只要世界观也会被塞主角。修复：user 消息只组装非空字段 + 显式声明"未提及维度勿强加设定"；数量支持 0=该节显式返回空数组；连兜底文案里的预设腔（"推进核心冲突并产生阵营后果"）都清了。升华一句：**发给模型的每个字节都是指令，包括你以为无害的空结构**。真实验证：一句话"深海玻璃修道院"+0 角色 0 任务，产出严格 0 NPC 0 任务（$0.0013）。

**Q：长上下文怎么省钱？**
A：三层缓存叙事——L1 prompt 哈希精确命中、L2 嵌入近邻（阈值 0.9，作用域=档位+system，防止不同接地互相污染）、DeepSeek 服务端前缀缓存按命中价计费（遥测单独记 cached tokens）。UI 动作层曾漏接缓存（每问必付），第十轮接入进程级共享缓存后重复问答实测 $0——这个"漏接"故事本身很好用：**缓存不是有没有，而是接没接到每条调用路径**。

## 13. Memory 答辩

**Q：你们的 agent 有记忆吗？**
A：分三层答。①调用级：无记忆，每次现场组装 context pack；②会话级：UI session_state（草稿/问答历史，随工作区切换整体清扫——曾修过 A 世界草稿可呈送进 B 世界的跨界泄漏）；③持久级：**文件即真相**（JSON 可 diff 可回滚），SQLite 只是可重建索引——两真相分离，误删库无损世界。**有意不做**对话式长期记忆：正典设定必须活在可审计的文件里，不能活在模型的暧昧记忆里。这个"不做"的理由比"做了"更显工程判断。

## 14. 通用性答辩（"换个说法就不行了吧？"）

A：分两层防。**生成侧**：prompt 全面去预设（中性设定者人设、题材/媒介/语言跟随 brief、零数量显式空数组、兜底文案不带题材腔），可选维度物理缺席；**理解侧**：专项清查三层召回——词面层泛化遍历每个对象的全部字符串字段（新内容类型自动纳入，无预设特征表）、LLM 逐项判定层吃下改写表达（强制 reason+原文 quote）、关系图扩散层抓"本体干净但邻居涉事"的对象。真实战绩：走私清查中，"灯塔议会"全文无走私字样，因"内部有人倒卖沉船遗物"被语义判定命中并给出原文引证。覆盖率诚实：判定上限外的对象计入 judge_skipped 且工作单显著标注——**宁可明示遗漏，不假装全覆盖**。

## 15. 排障方法论（Q11 的 Agent 加强版）

按"连接→解析→检索→成本→复现"五层，每层有探针有真实战例：
1. 连接层：探针 1 token 测连通；超时按任务分级（创世 240s——曾因一刀切 30s 让默认规模创世必死）；错误分类映射中文话术。
2. 解析层：真实模型 JSON 形状必漂移（实测 `timeline_order:"side"`、stage id 整数、rewards 对象）——归一化层矫正 + 每例一条回归测试，模式固定可复制。
3. 检索层：答非所问先看 context pack 的 refs 喂没喂对——先查输入再怪模型。
4. 成本层：遥测逐调用记账；"重复问答不为 $0"=缓存没接到该路径。
5. 复现层：offline 确定性替身全家桶，同输入必同输出，405 个 $0 测试守护回归；测试替身刻意与生产路径同契约（第十二轮就因替身还在解析旧 JSON 契约而抓出接口漂移）。
